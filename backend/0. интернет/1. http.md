# Обзор HTTP

https://developer.mozilla.org/en-US/docs/Web/HTTP/Overview

**HTTP** — это [протокол](https://developer.mozilla.org/en-US/docs/Glossary/Protocol) для получения ресурсов, таких как документы HTML. Это основа любого обмена данными в Интернете, и это клиент-серверный протокол, что означает, что запросы инициируются получателем, обычно веб-браузером. Полный документ обычно состоит из ресурсов, таких как текстовое содержимое, инструкции по макету, изображения, видео, скрипты и многое другое.

![[Pasted image 20241012152128 1.png]]

Клиенты и серверы взаимодействуют, обмениваясь отдельными сообщениями (в отличие от потока данных). Сообщения, отправленные клиентом, называются _запросами_ , а сообщения, отправленные сервером в качестве ответа, называются _ответами_ .

![[Pasted image 20241012152219 1.png]]

Разработанный в начале 1990-х годов, HTTP является расширяемым протоколом, который со временем развивался. Это протокол прикладного уровня, который отправляется по [TCP](https://developer.mozilla.org/en-US/docs/Glossary/TCP) или по зашифрованному [TLS](https://developer.mozilla.org/en-US/docs/Glossary/TLS) TCP-соединению, хотя теоретически можно использовать любой надежный транспортный протокол. Благодаря своей расширяемости он используется не только для извлечения гипертекстовых документов, но также изображений и видео или для отправки контента на серверы, как в случае с результатами HTML-форм. HTTP также может использоваться для извлечения частей документов для обновления веб-страниц по требованию.

## Компоненты систем на базе HTTP

HTTP — это клиент-серверный протокол: запросы отправляются одной сущностью, пользовательским агентом (или прокси от его имени). Чаще всего пользовательским агентом является веб-браузер, но это может быть что угодно, например, робот, который сканирует Интернет для заполнения и поддержания индекса поисковой системы.

Каждый отдельный запрос отправляется на сервер, который обрабатывает его и предоставляет ответ, называемый response _._ Между клиентом и сервером находится множество сущностей, совместно называемых [proxy](https://developer.mozilla.org/en-US/docs/Glossary/Proxy_server) , которые выполняют различные операции и действуют как шлюзы или [кэши](https://developer.mozilla.org/en-US/docs/Glossary/Cache) , например.

![[Pasted image 20241012152357 1.png]]

На самом деле, между браузером и сервером, обрабатывающим запрос, находится больше компьютеров: есть маршрутизаторы, модемы и многое другое. Благодаря многоуровневой структуре Интернета они скрыты в сетевых и транспортных уровнях. HTTP находится сверху, на прикладном уровне. Хотя он важен для диагностики сетевых проблем, базовые уровни в основном не имеют отношения к описанию HTTP.

### Клиент: пользовательский агент

Пользовательский _агент_ — это любой инструмент, который действует от имени пользователя. Эту роль в первую очередь выполняет веб-браузер, но ее также могут выполнять программы, используемые инженерами и веб-разработчиками для отладки своих приложений.

Браузер **всегда** является субъектом, инициирующим запрос. Он никогда не является сервером (хотя за эти годы были добавлены некоторые механизмы для имитации сообщений, инициированных сервером).

Чтобы отобразить веб-страницу, браузер отправляет исходный запрос на извлечение HTML-документа, представляющего страницу. Затем он анализирует этот файл, делая дополнительные запросы, соответствующие сценариям выполнения, информации о макете (CSS) для отображения и подресурсам, содержащимся на странице (обычно изображениям и видео). Затем веб-браузер объединяет эти ресурсы для представления полного документа, веб-страницы. Скрипты, выполняемые браузером, могут извлекать больше ресурсов на более поздних этапах, и браузер соответствующим образом обновляет веб-страницу.

Веб-страница — это гипертекстовый документ. Это означает, что некоторые части отображаемого контента являются ссылками, которые можно активировать (обычно щелчком мыши) для загрузки новой веб-страницы, что позволяет пользователю направлять своего user-agent и перемещаться по Интернету. Браузер преобразует эти указания в HTTP-запросы и далее интерпретирует HTTP-ответы, чтобы предоставить пользователю понятный ответ.

### Веб-сервер

На противоположной стороне канала связи находится сервер, который _обслуживает_ документ по запросу клиента. Сервер выглядит как виртуальная машина; но на самом деле это может быть набор серверов, разделяющих нагрузку (балансировка нагрузки), или другое программное обеспечение (такое как кэши, сервер базы данных или серверы электронной коммерции), полностью или частично генерирующее документ по запросу.

Сервер — это не обязательно одна машина, но на одной машине могут размещаться несколько экземпляров серверного ПО. С HTTP/1.1 и [`Host`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Host)заголовком они могут даже использовать один и тот же IP-адрес.

### Прокси

Между веб-браузером и сервером многочисленные компьютеры и машины передают HTTP-сообщения. Из-за многоуровневой структуры веб-стека большинство из них работают на транспортном, сетевом или физическом уровнях, становясь прозрачными на уровне HTTP и потенциально оказывая значительное влияние на производительность. Те, которые работают на уровнях приложений, обычно называются **прокси-серверами** . Они могут быть прозрачными, пересылая полученные запросы без их изменения каким-либо образом, или непрозрачными, в этом случае они каким-то образом изменят запрос перед его передачей на сервер. Прокси-серверы могут выполнять многочисленные функции:

- кэширование (кэш может быть публичным или частным, как кеш браузера)
- фильтрация (например, антивирусное сканирование или родительский контроль)
- балансировка нагрузки (чтобы позволить нескольким серверам обслуживать разные запросы)
- аутентификация (для контроля доступа к различным ресурсам)
- ведение журнала (позволяет хранить историческую информацию)

## Базовые аспекты HTTP

### HTTP — это просто

HTTP, как правило, разрабатывается так, чтобы быть простым и понятным человеку, даже с учетом дополнительной сложности, введенной в HTTP/2 путем инкапсуляции HTTP-сообщений во фреймы. HTTP-сообщения могут быть прочитаны и поняты людьми, что обеспечивает более простое тестирование для разработчиков и снижает сложность для новичков.

### HTTP расширяем

Представленные в HTTP/1.0, [заголовки HTTP](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers) позволяют легко расширять и экспериментировать с этим протоколом. Новые функциональные возможности могут быть даже введены простым соглашением между клиентом и сервером о семантике нового заголовка.

### HTTP не имеет состояния, но не сеансовый

HTTP не имеет состояния: нет связи между двумя запросами, последовательно выполняемыми в одном и том же соединении. Это сразу же может стать проблемой для пользователей, пытающихся взаимодействовать с определенными страницами последовательно, например, с помощью корзин покупок в электронной коммерции. Но в то время как ядро ​​самого HTTP не имеет состояния, файлы cookie HTTP позволяют использовать сеансы с состоянием. Используя расширяемость заголовков, файлы cookie HTTP добавляются в рабочий процесс, позволяя созданию сеанса в каждом запросе HTTP совместно использовать один и тот же контекст или одно и то же состояние.

### HTTP и соединения

Соединение контролируется на транспортном уровне и, следовательно, принципиально выходит за рамки HTTP. HTTP не требует, чтобы базовый транспортный протокол был основан на соединении; он только требует, чтобы он был _надежным_ или не терял сообщения (как минимум, выдавая ошибку в таких случаях). Среди двух наиболее распространенных транспортных протоколов в Интернете TCP является надежным, а UDP — нет. Поэтому HTTP полагается на стандарт TCP, который основан на соединении.

Прежде чем клиент и сервер смогут обменяться парой HTTP-запрос/ответ, они должны установить TCP-соединение, процесс, требующий нескольких циклов. Поведение HTTP/1.0 по умолчанию заключается в открытии отдельного TCP-соединения для каждой пары HTTP-запрос/ответ. Это менее эффективно, чем совместное использование одного TCP-соединения, когда несколько запросов отправляются в тесной последовательности.

Чтобы смягчить этот недостаток, HTTP/1.1 ввел _конвейеризацию_ (которая оказалась сложной в реализации) и _постоянные соединения_ : базовое TCP-соединение можно частично контролировать с помощью [`Connection`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Connection)заголовка. HTTP/2 пошел на шаг дальше, мультиплексируя сообщения по одному соединению, помогая поддерживать соединение теплым и более эффективным.

Продолжаются эксперименты по разработке лучшего транспортного протокола, более подходящего для HTTP. Например, Google экспериментирует с [QUIC](https://en.wikipedia.org/wiki/QUIC) , который основан на UDP, чтобы обеспечить более надежный и эффективный транспортный протокол.

## Что можно контролировать с помощью HTTP

Эта расширяемая природа HTTP со временем позволила обеспечить больший контроль и функциональность Интернета. Методы кэширования и аутентификации были функциями, обрабатываемыми на ранних этапах истории HTTP. Возможность ослабить _ограничение источника_ , напротив, была добавлена ​​только в 2010-х годах.

Вот список общих функций, которыми можно управлять с помощью HTTP:

- _[Кэширование](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching)_ : То, как кэшируются документы, может контролироваться HTTP. Сервер может давать указания прокси и клиентам о том, что кэшировать и на какой срок. Клиент может давать указания промежуточным кэширующим прокси игнорировать сохраненный документ.
- _Ослабление ограничения источника_ : для предотвращения слежки и других вторжений в личную жизнь веб-браузеры обеспечивают строгое разделение между веб-сайтами. Только страницы из **одного источника** могут получить доступ ко всей информации веб-страницы. Хотя такое ограничение является бременем для сервера, заголовки HTTP могут ослабить это строгое разделение на стороне сервера, позволяя документу стать лоскутным одеялом из информации, полученной из разных доменов; для этого могут быть даже причины, связанные с безопасностью.
- _Аутентификация_ : Некоторые страницы могут быть защищены, так что только определенные пользователи могут получить к ним доступ. Базовая аутентификация может быть предоставлена ​​HTTP, либо с использованием [`WWW-Authenticate`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/WWW-Authenticate)и подобных заголовков, либо путем установки определенного сеанса с использованием [HTTP cookie](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies) .
- _[Прокси и туннелирование](https://developer.mozilla.org/en-US/docs/Web/HTTP/Proxy_servers_and_tunneling)_ : серверы или клиенты часто располагаются в интрасетях и скрывают свой настоящий IP-адрес от других компьютеров. Затем HTTP-запросы проходят через прокси-серверы, чтобы преодолеть этот сетевой барьер. Не все прокси-серверы являются HTTP-прокси. Например, протокол SOCKS работает на более низком уровне. Другие протоколы, такие как ftp, могут обрабатываться этими прокси-серверами.
- _Сеансы_ : использование HTTP-cookie позволяет связывать запросы с состоянием сервера. Это создает сеансы, несмотря на то, что базовый HTTP является протоколом без состояния. Это полезно не только для корзин покупок в электронной коммерции, но и для любого сайта, позволяющего пользователю настраивать вывод.

## HTTP-поток

Когда клиент хочет связаться с сервером, будь то конечный сервер или промежуточный прокси-сервер, он выполняет следующие шаги:

1. Открыть TCP-соединение: TCP-соединение используется для отправки запроса или нескольких и получения ответа. Клиент может открыть новое соединение, повторно использовать существующее соединение или открыть несколько TCP-соединений с серверами.
    
2. Отправка HTTP-сообщения: HTTP-сообщения (до HTTP/2) читаются человеком. С HTTP/2 эти простые сообщения инкапсулируются в кадры, что делает невозможным их прямое чтение, но принцип остается тем же. Например:
    
    httpКопировать в буфер обмена
    
    ```
    GET / HTTP/1.1
    Host: developer.mozilla.org
    Accept-Language: fr
    ```
    
3. Прочитайте ответ, отправленный сервером, например:
    
    httpКопировать в буфер обмена
    
    ```
    HTTP/1.1 200 OK
    Date: Sat, 09 Oct 2010 14:28:02 GMT
    Server: Apache
    Last-Modified: Tue, 01 Dec 2009 20:18:22 GMT
    ETag: "51142bc1-7449-479b075b2891b"
    Accept-Ranges: bytes
    Content-Length: 29769
    Content-Type: text/html
    
    <!doctype html>… (here come the 29769 bytes of the requested web page)
    ```
    
4. Закройте или повторно используйте соединение для дальнейших запросов.
    

Если активирована HTTP-конвейеризация, можно отправлять несколько запросов, не дожидаясь полного получения первого ответа. HTTP-конвейеризация оказалась сложной для внедрения в существующих сетях, где старые части программного обеспечения сосуществуют с современными версиями. HTTP-конвейеризация была заменена в HTTP/2 более надежными мультиплексными запросами в кадре.

## HTTP-сообщения

HTTP-сообщения, как определено в HTTP/1.1 и более ранних версиях, являются читаемыми человеком. В HTTP/2 эти сообщения встроены в двоичную структуру, фрейм _,_ что позволяет проводить такие оптимизации, как сжатие заголовков и мультиплексирование. Даже если в этой версии HTTP отправляется только часть исходного HTTP-сообщения, семантика каждого сообщения остается неизменной, и клиент воссоздает (фактически) исходный HTTP/1.1-запрос. Поэтому полезно понимать HTTP/2-сообщения в формате HTTP/1.1.

Существует два типа HTTP-сообщений: запросы и ответы, каждый из которых имеет свой собственный формат.

### Запросы
Пример HTTP-запроса:

![Обзор HTTP-запроса GET с заголовками](https://mdn.github.io/shared-assets/images/diagrams/http/overview/http-request.svg)

Запросы состоят из следующих элементов:

- HTTP- [метод](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods) , обычно глагол типа [`GET`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/GET), [`POST`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/POST)или существительное типа [`OPTIONS`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/OPTIONS)или , [`HEAD`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods/HEAD)определяющее операцию, которую клиент хочет выполнить. Обычно клиент хочет получить ресурс (используя `GET`) или опубликовать значение [HTML-формы](https://developer.mozilla.org/en-US/docs/Learn/Forms) (используя `POST`), хотя в других случаях может потребоваться больше операций.
- Путь к ресурсу для извлечения; URL-адрес ресурса, очищенный от элементов, которые очевидны из контекста, например, без [протокола](https://developer.mozilla.org/en-US/docs/Glossary/Protocol) ( `http://`), [домена](https://developer.mozilla.org/en-US/docs/Glossary/Domain) (здесь, `developer.mozilla.org`) или [порта](https://developer.mozilla.org/en-US/docs/Glossary/Port) TCP (здесь, `80`).
- Версия протокола HTTP.
- Необязательные [заголовки](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers) , которые передают дополнительную информацию для серверов.
- Тело для некоторых методов, например `POST`, аналогичное тем, что находятся в ответах, содержащее отправленный ресурс.

### Ответы

Пример ответа:

![Обзор HTTP-ответа «200 OK» на запрос GET, включая заголовки ответа.](https://mdn.github.io/shared-assets/images/diagrams/http/overview/http-response.svg)

Ответы состоят из следующих элементов:

- Версия протокола HTTP, которой они следуют.
- Код [статуса](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status) , указывающий, был ли запрос успешным или нет, и почему.
- Сообщение о состоянии, неофициальное краткое описание кода состояния.
- HTTP- [заголовки](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers) , например, для запросов.
- При желании, тело, содержащее извлеченный ресурс.

## API на основе HTTP

Наиболее часто используемый API на основе HTTP — это [Fetch API](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API) , который можно использовать для выполнения HTTP-запросов из JavaScript. Fetch API заменяет API [`XMLHttpRequest`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest).

Другой API, [server-sent events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events) , представляет собой односторонний сервис, позволяющий серверу отправлять события клиенту, используя HTTP в качестве транспортного механизма. Используя [`EventSource`](https://developer.mozilla.org/en-US/docs/Web/API/EventSource)интерфейс, клиент открывает соединение и устанавливает обработчики событий. Клиентский браузер автоматически преобразует сообщения, поступающие в поток HTTP, в соответствующие [`Event`](https://developer.mozilla.org/en-US/docs/Web/API/Event)объекты. Затем он доставляет их обработчикам событий, зарегистрированным для событий, [`type`](https://developer.mozilla.org/en-US/docs/Web/API/Event/type "тип")если они известны, или обработчику [`onmessage`](https://developer.mozilla.org/en-US/docs/Web/API/EventSource/message_event "onmessage")событий, если не был установлен обработчик событий определенного типа.

## Заключение

HTTP — это расширяемый протокол, который прост в использовании. Клиент-серверная структура в сочетании с возможностью добавления заголовков позволяет HTTP развиваться вместе с расширенными возможностями Web.

Хотя HTTP/2 добавляет некоторую сложность, встраивая HTTP-сообщения в фреймы для повышения производительности, базовая структура сообщений осталась прежней с HTTP/1.0. Поток сеанса остается базовым, что позволяет исследовать и отлаживать его с помощью [сетевого монитора HTTP](https://firefox-source-docs.mozilla.org/devtools-user/network_monitor/index.html) .
